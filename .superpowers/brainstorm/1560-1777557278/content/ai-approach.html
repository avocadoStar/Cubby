AI 智能整理：两条路线对比

同样的用户体验，架构复杂度差10倍

文档里的方案（1.md + 5.md）

重型AI管线 — 过度设计
Bookmark → 提取metadata → OpenAI Embedding (1536维) → pgvector/Pinecone 存储 → K-Means/HDBSCAN 聚类 → GPT-4.1 生成文件夹结构 → Diff预览 → 用户确认 → 写入DB

适用场景

  • 10万+书签的语义搜索
  • 持续增量分类

对收藏夹项目的问题

  • 99%用户书签 < 1000条，embedding的价值趋近于零
  • 需要 pgvector 扩展或 Pinecone 服务，运维成本高
  • Embedding API 调用增加延迟和费用
  • 聚类参数调优是持续维护负担

我的推荐：轻量AI管线

直接LLM分类 — 够用且干净
Bookmark列表 [{title, url}] → LLM一次性分类 → 返回文件夹结构 → Diff预览 → 用户确认 → 写入DB

优势

  • 零新依赖 — 已有的 DashScope API 直接复用
  • 一次 API 调用完成全部分类
  • 无向量数据库、无聚类引擎、无新基础设施
  • LLM天然理解中英文混合书签
  • Prompt 调优比聚类参数调优简单10倍

代价

  • 单次输入有 token 上限（~1000条书签以内完全没问题）
  • 不支持实时增量语义搜索（Edge Clone 本来也不做这个）

核心逻辑

Embedding的价值在哪里？

Embedding 解决的是"海量数据中找相似"的问题 — 语义搜索、推荐系统、RAG。

书签自动整理解决的是"给100条链接分个类"的问题 — 这是LLM原生能力，不需要向量。

当你需要"在所有书签里搜索关于前端性能的内容"时再引入 embedding。那是搜索功能，不是整理功能。两个不同的需求，别混在一起。

结论：AI智能整理 = 用户点一个按钮 + 后端调一次LLM + 展示diff + 用户确认

就这么简单。embedding / vector DB / clustering 全部砍掉，等真需要语义搜索那天再加。
